# Requirements Document

## Introduction

This document covers four enterprise-grade frontend features for the StellarStream platform. The features span organization branding customization, performance stress testing, an executive health command center, and a real-time transaction activity feed. All features are built on the existing Next.js (App Router) + TypeScript stack, using Tailwind CSS and the established Stellar Glass design system.

---

## Glossary

- **Branding_Editor**: The UI component within the Settings page that allows organization administrators to configure their logo and primary color for public-facing surfaces.
- **Split_Link**: A public-facing payment link page generated by StellarStream for disbursement recipients.
- **PDF_Receipt**: A downloadable PDF document generated by StellarStream summarizing a completed disbursement.
- **CSS_Root**: The `:root` CSS custom properties block in `frontend/app/globals.css` that defines the Stellar Glass design token variables (`--stellar-primary`, etc.).
- **Live_Preview**: A real-time rendering of the Split_Link page appearance that updates as the user adjusts branding settings, without requiring a page reload.
- **Recipient_Grid**: The existing tabular UI component that renders a list of disbursement recipients with columns for address, label, amount, and token.
- **Recipient_Map**: The existing geographic map UI component that plots recipient locations as markers.
- **Stress_Test_Script**: The TypeScript script at `scripts/generate-dummy-recipients.ts` that generates dummy recipient rows for performance testing.
- **TTI**: Time to Interactive — the elapsed time from when the Recipient_Grid begins rendering until it responds to user input (scroll, click, keyboard).
- **Health_Command_Center**: The executive dashboard page that aggregates organization-wide disbursement pipeline health metrics.
- **Health_Card**: A UI card component displaying a single health metric (Success Rate, Total Volume, or Active Proposals) sourced from the Analytics Engine API.
- **Analytics_Engine_API**: The backend REST API (routed via `backend/src/api/analytics.routes.ts`) that provides aggregated disbursement statistics.
- **Global_Search**: The search bar component (`frontend/components/globalsearch.tsx`) that allows users to find records by TxHash or Recipient Name.
- **Transaction_Flow_Feed**: The real-time, vertically scrolling activity log component placed in the dashboard sidebar that displays organization-wide actions.
- **Event_Scraper**: The backend service that emits organization activity events over WebSockets (Socket.IO) to connected frontend clients.
- **Log_Feed**: The visual container within the sidebar that renders Transaction_Flow_Feed items in reverse-chronological order.
- **Pulsing_Icon**: An animated icon (using the existing `animate-pulse-border` CSS class) applied to feed items whose status is `ledger_confirmation` to indicate an in-progress on-chain confirmation.
- **Organization_Admin**: A user with the organization administrator role who has permission to modify branding and view the Health_Command_Center.
- **Developer**: A user who runs the Stress_Test_Script locally to validate component performance.

## Requirements

### Requirement 1: Enterprise Customization — Branding Tab (#1006)

**User Story:** As an Organization_Admin, I want to upload my organization's logo and set a primary brand color in a dedicated Branding tab within Settings, so that public Split_Links and PDF_Receipts reflect my organization's visual identity.

#### Acceptance Criteria

1. THE Settings_Page SHALL render a "Branding" tab alongside the existing "Gas Management" and "Security & Privacy" sections, accessible at `/dashboard/settings`.

2. WHEN an Organization_Admin navigates to the Branding tab, THE Branding_Editor SHALL display an image upload control, a color picker input, and a Live_Preview panel.

3. WHEN an Organization_Admin selects an image file via the upload control, THE Branding_Editor SHALL accept files of type PNG, JPG, or SVG with a maximum file size of 2 MB.

4. IF an Organization_Admin uploads a file that exceeds 2 MB or is not of type PNG, JPG, or SVG, THEN THE Branding_Editor SHALL display an inline validation error message describing the constraint that was violated.

5. WHEN an Organization_Admin selects a valid logo file, THE Live_Preview SHALL render the uploaded logo image within the Split_Link preview panel within 300 ms of file selection.

6. WHEN an Organization_Admin changes the primary color value using the color picker, THE Branding_Editor SHALL update the `--stellar-primary` CSS custom property on the `:root` element in real time, causing the Live_Preview to reflect the new color without a page reload.

7. WHEN an Organization_Admin changes the primary color value, THE Live_Preview SHALL update its rendered appearance within 100 ms of the color input change event.

8. WHEN an Organization_Admin clicks the "Save Branding" button, THE Branding_Editor SHALL persist the logo URL and primary color value to the organization's settings store.

9. IF the save operation fails due to a network error, THEN THE Branding_Editor SHALL display an error notification and retain the unsaved values in the form fields.

10. WHEN an Organization_Admin returns to the Branding tab after a successful save, THE Branding_Editor SHALL pre-populate the color picker and logo preview with the previously saved values.

11. WHERE the organization has not yet configured branding, THE Branding_Editor SHALL display the default `--stellar-primary` value (`#00f5ff`) in the color picker and a placeholder logo prompt in the upload control.

### Requirement 2: "X-Ray" Component Stress Test (#1007)

**User Story:** As a Developer, I want to run an automated stress test against the Recipient_Grid and Recipient_Map with 500 or more recipients, so that I can verify the browser remains stable and measure Time to Interactive to identify and fix unnecessary re-renders.

#### Acceptance Criteria

1. THE Stress_Test_Script SHALL generate a minimum of 500 unique dummy recipient records, each containing the fields: `id`, `address`, `label`, `amount`, `token`, `taxId` (optional), `transactions`, and `lastActive`.

2. WHEN the Stress_Test_Script is executed, THE Stress_Test_Script SHALL write the generated records to `frontend/public/dummy-recipients.json` and `frontend/public/dummy-recipients.csv`.

3. WHEN the Recipient_Grid renders 500 or more recipient rows loaded from `dummy-recipients.json`, THE Recipient_Grid SHALL reach a TTI of 3000 ms or less as measured from the start of the render call.

4. WHILE the Recipient_Grid is rendering 500 or more rows, THE Browser SHALL not throw an unhandled JavaScript exception or crash.

5. WHEN a recipient row component's props have not changed between renders, THE Recipient_Grid SHALL not re-render that row component, enforced via `React.memo` wrapping on the row component.

6. WHEN the Recipient_Map renders 500 or more recipient markers loaded from `dummy-recipients.json`, THE Recipient_Map SHALL reach a TTI of 3000 ms or less as measured from the start of the render call.

7. WHILE the Recipient_Map is rendering 500 or more markers, THE Browser SHALL not throw an unhandled JavaScript exception or crash.

8. THE Stress_Test_Script SHALL log the count of generated records and the output file paths to the console upon successful completion.

9. IF the `@faker-js/faker` dependency is not installed, THEN THE Stress_Test_Script SHALL exit with a non-zero status code and print an actionable error message to stderr.

### Requirement 3: "Organization-Health" Command Center (#1009)

**User Story:** As an Organization_Admin, I want a high-level dashboard that shows the health of my disbursement pipeline across all assets, so that I can quickly assess operational status and find any past disbursement by transaction hash or recipient name.

#### Acceptance Criteria

1. THE Health_Command_Center SHALL be accessible as a dedicated page or prominent section within the dashboard at a route reachable from the main navigation.

2. WHEN the Health_Command_Center page loads, THE Health_Card SHALL render three metric tiles: "Success Rate %", "Total Volume (30d)", and "Active Proposals".

3. WHEN the Health_Command_Center page loads, THE `useOrganizationHealth` hook SHALL fetch aggregated data from the Analytics_Engine_API endpoint and supply it to the Health_Card component.

4. WHILE the Analytics_Engine_API request is in flight, THE Health_Command_Center SHALL display a loading skeleton in place of each Health_Card metric tile.

5. IF the Analytics_Engine_API returns an error response, THEN THE Health_Command_Center SHALL display an inline error state within the affected Health_Card tile, including a "Retry" action.

6. WHEN the Analytics_Engine_API returns data, THE Health_Card SHALL display the `successRate` value as a percentage formatted to one decimal place (e.g., "98.5%").

7. WHEN the Analytics_Engine_API returns data, THE Health_Card SHALL display the `totalVolume30d` value formatted as a USD equivalent using the existing `formatUsdValue` utility.

8. WHEN the Analytics_Engine_API returns data, THE Health_Card SHALL display the `activeProposals` value as an integer count.

9. THE Health_Command_Center SHALL render the Global_Search component, allowing users to search for past disbursements by TxHash or Recipient Name.

10. WHEN a user enters a query of 3 or more characters into the Global_Search component, THE Global_Search SHALL display matching results filtered from the disbursement index within 300 ms of the last keystroke.

11. WHEN a user selects a search result from the Global_Search dropdown, THE Health_Command_Center SHALL navigate the user to the detail view for the selected disbursement record.

12. IF a search query returns no matching disbursement records, THEN THE Global_Search SHALL display a "No results found" empty state message.

13. WHEN the Health_Command_Center data was last fetched more than 5 minutes ago, THE Health_Command_Center SHALL automatically re-fetch data from the Analytics_Engine_API to refresh the displayed metrics.

### Requirement 4: "Transaction-Flow" Live Feed (#1010)

**User Story:** As a dashboard user, I want a real-time, scrolling activity feed in the sidebar that shows all organization actions as they happen, so that I can monitor disbursement activity without leaving the dashboard.

#### Acceptance Criteria

1. THE Log_Feed SHALL be rendered inside the dashboard sidebar (`frontend/components/dashboard/sidebar.tsx`) below the existing navigation items, replacing or extending the current `TransactionFeed` component.

2. WHEN the dashboard loads, THE `useTransactionFeed` hook SHALL establish a WebSocket connection to the Event_Scraper using Socket.IO and begin receiving activity events.

3. WHEN the Event_Scraper emits a `transaction-event`, THE Log_Feed SHALL prepend the new event item to the top of the feed within 200 ms of receipt.

4. THE Log_Feed SHALL retain a maximum of 50 feed items in memory at any time; WHEN a new item is received that would exceed this limit, THE Log_Feed SHALL remove the oldest item.

5. WHEN a feed item has a `type` value of `recipient_added`, THE Log_Feed SHALL render the item with a description in the format: "[actor] added a recipient".

6. WHEN a feed item has a `type` value of `split_approved`, THE Log_Feed SHALL render the item with a description in the format: "[asset] Split Approved".

7. WHEN a feed item has a `status` value of `ledger_confirmation`, THE Log_Feed SHALL render a Pulsing_Icon on that item using the `animate-pulse-border` CSS animation class.

8. WHEN the Event_Scraper emits a `ledger-confirmation` event for an existing feed item `id`, THE Log_Feed SHALL update that item's status in place rather than inserting a duplicate entry.

9. IF the WebSocket connection to the Event_Scraper is lost, THEN THE `useTransactionFeed` hook SHALL attempt to reconnect up to 5 times with exponential back-off, and THE Log_Feed SHALL display a "Reconnecting…" status indicator during each attempt.

10. IF all 5 reconnection attempts fail, THEN THE Log_Feed SHALL display a "Connection lost — refresh to retry" error state.

11. WHEN the sidebar is in its collapsed state, THE Log_Feed SHALL be hidden to preserve layout space, and SHALL become visible again when the sidebar is expanded.

12. WHEN a feed item is rendered, THE Log_Feed SHALL display the item's `timestamp` formatted as a relative time string (e.g., "2m ago") using the existing `formatTimestamp` helper pattern.

13. THE Log_Feed SHALL be keyboard-accessible, with each feed item reachable via Tab key and readable by screen readers via appropriate ARIA roles and labels.
